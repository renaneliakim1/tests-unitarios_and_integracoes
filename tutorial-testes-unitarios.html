Testes Unitários com Jest (JS) e unittest (Python)

Fundamentos

O que é teste unitário?

O teste unitário verifica uma unidade pequena (função/método) de forma isolada.

Padrão AAA

O padrão AAA (Arrange, Act, Assert) é uma convenção para escrever testes unitários.

  • Arrange: Configura o estado necessário para o teste, como criar objetos, definir variáveis e configurar mocks.
  • Act: Executa a unidade de código sendo testada (função/método).
  • Assert: Verifica se o resultado obtido é o esperado, usando asserções (ex: expect em Jest, assert em unittest).

Propriedades:

  • Determinismo: O mesmo input (entrada) sempre produz o mesmo output (saída).
  • Repetibilidade: O teste pode ser executado várias vezes sem alterar o resultado.
  • Independência: Cada teste deve ser capaz de ser executado independentemente de outros testes.

Cobertura:

Mede quais linhas/branches são exercitadas; indicador, não objetivo único.

Testes para adição

JavaScript (Jest) 

// Importando a classe Calculadora do arquivo calculadora.js
const { Calculadora } = require('./calculadora');

// Criando uma instância da classe Calculadora
const calc = new Calculadora();

// Teste para soma de dois inteiros
it('soma dois inteiros', () => {
  expect(calc.adicao(2, 3)).toBe(5);
});

// Teste para soma com zero
it('soma com zero', () => {
  expect(calc.adicao(0, 7)).toBe(7);
});

// Teste para soma de números negativos
it('soma números negativos', () => {
  expect(calc.adicao(-2, -3)).toBe(-5);
});

Python (unittest) 

# Importando o pacote de testes unitários
import unittest

# Importando a classe Calculadora do arquivo calculadora.py
from calculadora import Calculadora

# Criando a classe de testes para a Calculadora
class TestCalculadora(unittest.TestCase):

  # Função de teste para soma de dois inteiros
  def test_soma_inteiros(self):
    calc = Calculadora()
    self.assertEqual(calc.adicao(2, 3), 5)
    
  # Função de teste para soma com zero
  def test_soma_com_zero(self):
    calc = Calculadora()
    self.assertEqual(calc.adicao(0, 7), 7)

  # Função de teste para soma de números negativos  
  def test_soma_negativos(self):
    calc = Calculadora()
    self.assertEqual(calc.adicao(-2, -3), -5)

# Execução dos testes
if __name__ == '__main__':
  unittest.main()

Executando os testes

  • JavaScript: npm test, npm run test ou jest
  • Python: python -m unittest ou python -m unittest discover -s tests

Desafios

  • JavaScript:
    • Implementar subtracao(a,b), multiplicacao(a,b), divisao(a,b) e criar testes.
    • Casos de borda: zeros, negativos, decimais, divisão por zero (decidir comportamento: erro, Infinity, ou retorno customizado).
  • Python:
    • Implementar os mesmos métodos e criar testes equivalentes.
    • Casos de borda: ZeroDivisionError (assertar que a exceção é lançada), precisão de decimais.

Boas Práticas

  • Nomear testes de forma clara e específica.
  • Um cenário por teste; evitar dependência entre testes.
  • Cobrir casos típicos e extremos (zero, negativo, decimais).
  • Em divisão, decidir e documentar comportamento esperado para divisor zero.
  • Usar toBeCloseTo/assertAlmostEqual para floating point.
  • Automatizar execução no CI e gerar cobertura:
    • Jest: npm test --coverage
    • Python: coverage run -m unittest e coverage report (via coverage.py)

Cobertura de Testes (JavaScript e Python)

A cobertura de testes (test coverage) mede quanto do seu código está sendo exercitado pelos testes automatizados. Ela não garante qualidade por si só, mas é um ótimo indicador de onde você pode ter riscos ou código não testado.

Objetivo da cobertura de testes

  1. Identificar partes não testadas: Mostra quais funções, linhas ou branches (caminhos condicionais) não foram executados pelos testes.
  2. Reduzir bugs em produção: Quanto maior (e melhor) a cobertura, menor a chance de erros passarem despercebidos.
  3. Ajudar na manutenção: Código bem coberto é mais seguro de refatorar.
  4. Aumentar confiança no sistema: Especialmente importante em sistemas críticos.

Tipos de métricas comumente avaliadas

  • Statements / Stmts: linhas de código executadas.
  • Branches: caminhos condicionais explorados (ifs, elses, switches).
  • Functions / Funcs: funções chamadas e executadas.
  • Lines: total de linhas de fato percorridas.

Como usar cobertura nas nossas linguagens

1. Instalação e Preparação

  • JavaScript (Jest): O pacote principal do Jest já possui as análises de cobertura embutidas.
  • Python: Para obter relatórios equivalentes aos do Jest, instale o pacote coverage.
    pip install coverage

2. Rodar testes com geração de relatórios

  • JavaScript:

    npx jest --coverage
# ou usando o script caso configurado: npm test -- --coverage

  • Python:
    Para avaliar, primeiro rode os testes através do coverage run e depois verifique com o coverage report:

    coverage run -m unittest discover
coverage report -m

3. Visualizando a cobertura gerada

No terminal:
A saída indicará estatísticas tabulares por arquivo contendo as porcentagens para cada métrica listada.

Relatório HTML (mais útil):

  • JavaScript: O Jest cria automaticamente uma pasta coverage/lcov-report/index.html.
  • Python: Rode o comando coverage html e abra a pasta gerada htmlcov/index.html.

Em ambos, ao abrir no navegador, você consegue ver o código colorido:

  • Verde = código coberto pelo teste.
  • Vermelho = lógica ou linha não testada.

Configuração opcional de metas

É possível configurar metas mínimas. O teste vai falhar se essas métricas de cobertura não forem atingidas, bloqueando o envio de código inacabado para a base principal.

Em JavaScript (package.json ou jest.config.js):

module.exports = {
  collectCoverage: true,
  coverageThreshold: {
    global: {
      branches: 80,
      functions: 80,
      lines: 80,
      statements: 80
    }
  }
};

Em Python (no arquivo .coveragerc):

[report]
fail_under = 80

Boas práticas e Dicas

  • Cobertura alta ≠ testes bons.
  • Foque em testar comportamento e os casos limite, não se importe apenas em tornar as linhas "verdes".
  • Se você vir cobertura baixa em branches, normalmente faltam testes para alguma condicional (if/else) ou algum mecanismo de erro (ex: try...catch)

Desafio de Cobertura Parcial na Calculadora

Imagine que você resolveu o Desafio apontado na seção anterior e sua classe foi adaptada nos testadores Python e JavaScript para incluir: adicao, subtracao, multiplicacao, divisao.

Mas, na pressa, os testes que você escreveu foram feitos apenas para adição e subtração. O que a cobertura vai mostrar?

No terminal e nos relatórios as métricas irão mostrar (aproximadamente, no arquivo calculadora):

File          | % Stmts | % Branch | % Funcs | % Lines
--------------|---------|----------|---------|--------
calculadora   | 50%     | 100%     | 50%     | 50%

Métricas Explicadas:

  • Functions: ~50% (apenas 2 das 4 funções foram testadas).
  • Statements / Lines: ~50% (as lógicas de multiplicação e divisão nunca foram disparadas).
  • Branches: 100% caso nenhuma das suas 4 funções aplique estruturas condicionais (ex: validando divisão por zero).

Você verá os métodos de soma e subtração pintados de verde, mas de vermelho a multiplicação e a divisão. A cobertura avisa e ilustra muito bem que: "as funções existem, mas nunca rodaram nos testes" ajudando a apontar quais recursos podem, potencialmente, abrigar lógicas defeituosas em produção originados de funcionalidades não devidamente avaliadas ou esquecidas.

Tratamento de Erros e Exceções

Na seção anterior, vimos como usar testes para cobrir fluxos esperados (ex: soma de números). Agora, vamos focar em garantir que sua aplicação reaja corretamente quando algo inesperado acontecer, como uma divisão por zero ou entrada de dados inválida.

O objetivo principal aqui é verificar se o tratamento de erros que você implementou (geralmente try...catch ou try...except) funciona conforme o esperado, lançando a exceção correta ou retornando o valor de erro esperado, e se o código que consome essa função sabe lidar com isso.

Por que testar erros?

Testar o "caminho infeliz" (unhappy path) é tão importante quanto o "caminho feliz". Isso garante que o sistema não quebre de forma catastrófica e forneça feedback útil.

1. JavaScript (Jest): Usando .toThrow()

No Jest, para testar se uma função lança um erro, usamos o matcher .toThrow().
Importante: Você deve passar uma função para o expect, e não o resultado da execução da função, para que o Jest possa capturar a exceção.

Exemplo de implementação (calculadora.js): 

divisao(a, b) {
  if (b === 0) {
    throw new Error("Divisão por zero");
  }
  return a / b;
}

Exemplo de teste (calculadora.test.js): 

it('deve lançar erro ao dividir por zero', () => {
  const calc = new Calculadora();
  // Usamos uma arrow function para que o Jest execute o código e capture o erro
  expect(() => calc.divisao(10, 0)).toThrow("Divisão por zero");
});

2. Python (unittest): Usando assertRaises

No Python, usamos o método self.assertRaises() dentro de um gerenciador de contexto (with). O teste passará se o código dentro do bloco with lançar a exceção especificada.

Exemplo de implementação (calculadora.py): 

def divisao(self, a, b):
    if b == 0:
        raise ValueError("Divisão por zero")
    return a / b

Exemplo de teste (test_calculadora.py): 

def test_divisao_por_zero(self):
    calc = Calculadora()
    # O teste verifica se o erro ValueError é disparado
    with self.assertRaises(ValueError) as context:
        calc.divisao(10, 0)
    
    # Opcional: Validar se a mensagem de erro está correta
    self.assertEqual(str(context.exception), "Divisão por zero")

Desafio

  • crie testes para validar erros em operações com valores não numéricos

Atividade para casa: cálculo do Índice de Massa Corporal (IMC)

Nesta atividade, você deverá aplicar todos os conceitos vistos: criação de métodos, testes unitários, tratamento de exceções e análise de cobertura podendo ser realizada tanto em JavaScript quanto em Python.

O objetivo é criar uma classe que gerencie o cálculo e a classificação do Índice de Massa Corporal (IMC).

O Problema

O IMC é uma medida internacional usada para calcular se uma pessoa está no peso ideal. A fórmula é:
IMC = peso / (altura²) ou IMC = peso / (altura * altura)

Requisitos da Implementação

Crie uma classe chamada CalculadoraIMC (ou similar) com os seguintes métodos:

  1. calcular(peso, altura):

    • Deve receber o peso (kg) e a altura (m).
    • Deve retornar o valor do IMC calculado.
    • Validação: Lance um erro (ex: ValueError ou Error) caso o peso ou altura sejam menores ou iguais a zero.
    • Validação: Lance um erro caso as entradas não sejam numéricas.

  2. classificar(imc):

    • Deve receber o valor do IMC e retornar uma string com a classificação:
      • Menor que 18.5: "Abaixo do peso"
      • 18.5 a 24.9: "Peso normal"
      • 25.0 a 29.9: "Sobrepeso"
      • 30.0 ou mais: "Obesidade"

Requisitos dos Testes

Escreva uma suíte de testes (Jest para JS ou unittest para Python) que cubra:

  • Caminho Feliz: Teste pelo menos um cenário para cada categoria da tabela acima.
  • Limites (Boundary Testing): Teste os valores exatos de transição (ex: 18.5, 24.9, 25.0).
  • Tratamento de Erros: Teste se as exceções são lançadas corretamente ao passar:
    • Peso zero ou negativo.
    • Altura zero ou negativa.
    • Strings ou valores nulos no lugar de números.
  • Cobertura: Gere o relatório e garanta que 100% das linhas e caminhos (branches) do código foram testados.
