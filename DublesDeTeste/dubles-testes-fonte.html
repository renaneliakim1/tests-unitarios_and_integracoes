Dominando os Dublês de Testes

1. Introdução Teórica: O Porquê dos Dublês

Vejamos a analogia clássica: em um filme, um dublê substitui o ator em cenas perigosas ou caras. No código, o dublê de teste (Test Double) substitui uma parte do sistema que é difícil, lenta ou impossível de lidar durante a execução dos testes automatizados, como:

  • Banco de dados (não queremos alterar dados reais)
  • APIs externas (podem estar fora do ar ou cobrar por requisições)
  • Sistema de arquivos (leitura e escrita lentas)

Os 5 Tipos de Dublês

Os dublês não são todos iguais. Cada um tem um propósito específico. Veja a tabela comparativa:

Tipo Função Principal Analogia
Dummy Objeto passado apenas para preencher parâmetros necessários na chamada de uma função, mas que nunca será realmente utilizado. Um manequim no fundo da cena, apenas para compor o cenário.
Stub Objeto que fornece respostas prontas e pré-configuradas para as chamadas feitas durante o teste. Ele não responde a nada além do que foi programado. Um ator figurante que só sabe dizer uma única fala decorada: "Sim" ou "Não".
Spy Um "Stub" que também grava informações sobre como foi chamado (quais argumentos recebeu, quantas vezes foi invocado). Muito útil para verificar se uma função chamou outra corretamente. Um espião infiltrado que anota quem entrou na sala e o que disse.
Mock Objeto pré-programado com expectativas estritas de como deve ser usado. Se o mock não for chamado da forma exata que foi configurado (ex: com os parâmetros certos, o número certo de vezes), o teste falha. Um roteiro rigoroso de teatro: "Você deve falar a frase X no tempo exato Y, senão a cena corta".
Fake Uma implementação de fato funcional, mas simplificada ou "pequena" para ser usada no teste (ex: um banco de dados em memória, em vez de um PostgreSQL real). Um cenário de papelão que funciona como uma porta de verdade, mas não leva a lugar nenhum na vida real.

Entendendo a diferença: Mock Objects vs. Dados Mockados

É muito comum no dia a dia desenvolvedores dizerem "vamos mockar esses dados", mas na teoria rigorosa de testes, eles estão frequentemente criando Stubs ou Dummies. Entender a diferença é o segredo para dominar testes:

1. Dados Mockados (Foco no ESTADO - O que a função precisa ou retorna)

Isso é o que chamamos de Stub ou Dummy. O objetivo aqui é apenas "enganar" a função para que ela funcione sem quebrar, fornecendo a ela as informações falsas que ela exige. Você não testa o objeto falso em si.

Exemplo Didático:

// O Dummy/Stub: Apenas um objeto com dados falsos para a função não dar erro
const usuarioFalso = { nome: 'João', idade: 25 }; 

// O foco do teste é no resultado da função principal, e não no usuarioFalso (Estado)
const desconto = calcularDesconto(usuarioFalso);
expect(desconto).toBe(10); // "Dado esse usuário, o desconto gerado é 10?"

2. Mocks (Foco no COMPORTAMENTO - O que a função faz)

O Mock Object (ou Spy) é mais "inteligente". Você o cria não apenas para fornecer dados, mas para vigiar como o seu código interage com ele. Você o usa quando precisa testar a comunicação entre sistemas.

Exemplo Didático:

// O Mock: Um objeto falso que "grava" o que acontece com ele
const emailServiceMock = {
  enviarBoasVindas: jest.fn() // O Jest cria uma função espiã aqui
};

// Ao rodar a nossa função principal, ela deve usar o serviço de e-mail por debaixo dos panos
cadastrarUsuario('Maria', emailServiceMock);

// Aqui o teste não é sobre um valor retornado, mas sobre a AÇÃO (Comportamento)
expect(emailServiceMock.enviarBoasVindas).toHaveBeenCalledWith('Maria'); 
// "A função de cadastro lembrou de chamar o envio de e-mail passando 'Maria'?"

Resumo da Regra de Ouro:

  • Quer testar o que foi gerado/retornado? Use um Stub/Dummy (foco no estado).
  • Quer testar como algo foi chamado/executado? Use um Mock/Spy (foco no comportamento).

2. Preparação do Ambiente

Para entendermos na prática, vamos iniciar um projeto simples com Node.js e a biblioteca de testes Jest.

Execute os comandos abaixo no seu terminal:

# Cria o diretório do projeto e entra nele
mkdir testes-dubles && cd testes-dubles

# Inicializa o projeto Node.js com configurações padrão
npm init -y

# Instala o Jest (framework de testes) e o Axios (para requisições) como dependências de desenvolvimento
npm install --save-dev jest
npm install axios

Estrutura de Pastas e Arquivos

Após criar o projeto, organize as pastas e arquivos da seguinte forma. Você pode criar os arquivos vazios agora e preenchê-los conforme avançamos no tutorial:

testes-dubles/

│   ├── githubService.js  # Serviço que consome a API do GitHub

"scripts": {

3. Prática Parte I: Stubs e Spies

Cenário: Temos uma função de finalizar compra que envia um e-mail. Não queremos disparar e-mails de verdade toda vez que rodarmos o teste, pois isso lotaria a caixa de entrada dos clientes e tornaria o teste lento.

O Código (src/orderService.js) 

// Simula um serviço de envio de e-mails
const emailService = {
  send: (msg) => { 
    // Imagine uma conexão demorada e pesada com um servidor SMTP aqui
    console.log(`Enviando e-mail real: ${msg}`);
  }
};

// Função principal que queremos testar
function checkout(cart, email) {
  // Regra: se o carrinho estiver vazio, não finaliza a compra
  if (cart.length === 0) return false;
  
  // Chama o serviço de e-mail (dependência externa)
  emailService.send(`Pedido confirmado para ${email}`);
  
  // Retorna sucesso
  return true;
}

// Exporta as funções para serem usadas no teste
module.exports = { checkout, emailService };

O Teste (tests/order.test.js)

Vamos usar o jest.spyOn para criar um Spy que intercepta a chamada de e-mail e também funciona como Stub (retornando true imediatamente em vez de executar o código original).

// Importando o módulo a ser testado e suas dependências
const { checkout, emailService } = require('../src/orderService');

test('Deve chamar o envio de e-mail ao finalizar compra com sucesso', () => {
  // Criando um Spy/Stub na função 'send' do 'emailService'
  // Ele vai interceptar a chamada e retornar "true" de forma silenciosa
  const spy = jest.spyOn(emailService, 'send').mockImplementation(() => true);

  // Executando a função que estamos testando
  // Obs: passamos um e-mail genérico para o teste
  const result = checkout(['item1', 'item2'], 'user@exemplo.com');

  // Verifica se o comportamento interno da função ocorreu como o esperado
  expect(result).toBe(true);
  
  // Verifica se o Spy foi chamado contendo o e-mail correto (foco no COMPORTAMENTO - Mock/Spy)
  expect(spy).toHaveBeenCalledWith(expect.stringContaining('user@exemplo.com'));
  
  // Boas práticas: Limpa o mock após o uso para não afetar outros testes
  spy.mockRestore(); 
});

4. Prática Parte II: Mocks com APIs Externas

Este é o caso de uso mais comum no mercado de trabalho: simular chamadas de rede com bibliotecas como axios ou a API nativa fetch.

Cenário: Criar uma função que busca os dados de um usuário na API pública do GitHub. Não queremos fazer uma requisição HTTP real no teste, pois precisamos que o teste seja rápido e não dependa de internet.

O Código (src/githubService.js) 

// Importa o axios para fazer requisições HTTP
const axios = require('axios');

// Função assíncrona que busca os dados na API
async function getUserData(username) {
  // Faz a requisição GET real para o GitHub
  const response = await axios.get(`https://api.github.com/users/${username}`);
  
  // Retorna apenas a carga de dados (data) da resposta
  return response.data;
}

module.exports = { getUserData };

O Teste (tests/github.test.js)

Aqui, vamos mockar o módulo inteiro do Axios. O Jest substituirá as funções originais do Axios por simulações controláveis.

// Importa o axios (que será mockado)
const axios = require('axios');
// Importa a função que vamos testar
const { getUserData } = require('../src/githubService');

// Informa ao Jest para mockar todo o módulo 'axios'
jest.mock('axios');

test('Deve retornar dados do usuário mockados sem fazer requisição real', async () => {
  // 1. Preparação (Arrange): Criação do nosso STUB (Dados Mockados)
  const mockUser = { name: 'Freedson', bio: 'Developer' };
  
  // Configura o comportamento do Mock (O que o axios.get vai retornar quando for chamado)
  // Como axios.get retorna uma Promise, usamos mockResolvedValue
  axios.get.mockResolvedValue({ data: mockUser });

  // 2. Ação (Act): Chamamos a nossa função
  const data = await getUserData('freedson');
  
  // 3. Verificação (Assert): Comparamos os resultados
  expect(data.name).toBe('Freedson'); // Testa o retorno
  
  // Verifica o COMPORTAMENTO: Garante que a API foi chamada exatamente 1 vez
  expect(axios.get).toHaveBeenCalledTimes(1);
  
  // Verifica se foi chamado com a URL correta
  expect(axios.get).toHaveBeenCalledWith('https://api.github.com/users/freedson');
});

5. Desafio Prático

Agora é a sua vez! Abaixo, temos um código que foi escrito de uma forma que dificulta a realização de testes unitários limpos. Ele processa pagamentos e possui regras de negócio embutidas ao gateway.

O Arquivo: src/paymentGate.js

// Gateway de pagamentos simulado (difícil de testar sem mocks)
const gateway = {
  authorize: (amount) => {
    // Imagina que isso chama uma API bancária real e cobra dinheiro de verdade
    console.log(`[ALERTA] Processando cobrança real de R$${amount}!`);
    return { status: 'authorized', transactionId: Math.random() };
  }
};

// Função de negócio acoplada ao gateway de forma direta
function processPayment(amount) {
  if (amount <= 0) {
    throw new Error('Valor inválido');
  }

  // Regra de negócio: pagamentos acima de 1000 reais exigem análise manual, 
  // então NÃO devem ir para o gateway.
  if (amount > 1000) {
    return 'pending_manual_review';
  }

  // Se for <= 1000, autoriza diretamente no gateway
  const response = gateway.authorize(amount);
  
  if (response.status === 'authorized') {
    return 'success';
  }
  
  return 'failed';
}

module.exports = { processPayment, gateway };

O que você deve fazer:

Crie o arquivo tests/payment.test.js e escreva dois testes para a função processPayment:

  1. Teste 1: Se o valor for 1500 (maior que 1000), o teste deve garantir que a função retorna 'pending_manual_review' E que o método gateway.authorize não foi chamado em hipótese alguma (use um Spy para verificar isso).
  2. Teste 2: Se o valor for 500 (menor ou igual a 1000), o teste deve mockar o gateway.authorize para retornar uma resposta de sucesso simulada ({ status: 'authorized' }) e garantir que a função retorna 'success'.

Dica: Use jest.spyOn(gateway, 'authorize'). Lembre-se de usar o método mockClear() do seu spy entre um teste e outro para evitar que as contagens de chamadas se misturem!

Considerações Finais

  • Mocks e Stubs testam coisas diferentes: Mocks validam a comunicação entre os componentes (comportamento). Stubs validam a lógica interna baseada em um cenário controlado (estado).
  • Cuidado com o Excesso de Mocks: Avise que mockar tudo pode esconder bugs de integração cruciais. Se você mockar o banco de dados e a API ao mesmo tempo, você só estará testando a sua imaginação, e não se os componentes realmente conversam bem na vida real.
