🎯 Testes Automatizados (JavaScript e Python)

Este guia foi desenvolvido para transformar o conceito de "testar" em uma habilidade prática de engenharia de software. Vamos explorar como sair da validação manual e repetitiva para a construção de uma rede de segurança de código confiável.

🧠 1. Introdução

📌 O que são Testes Automatizados?

Diferente de um teste manual, onde um humano interage com o sistema para verificar se ele funciona, os testes automatizados são códigos criados para validar outros códigos.

Imagine que você tem um robô assistente: toda vez que você altera uma linha de código no seu projeto, esse robô executa milhares de verificações em segundos para garantir que você não quebrou nada que já estava funcionando.

O Fluxo da Automação:

  1. Entrada de Dados: O teste fornece valores específicos (ex: números para uma soma).
  2. Execução da Lógica: O código do sistema processa esses valores.
  3. Validação (Asserção): O teste compara o resultado obtido com o resultado esperado. Se forem diferentes, o teste falha.
Note

Observação Didática (Insight): Automatizar não é apenas "rodar ferramentas". É transformar o conhecimento sobre como o sistema deve se comportar em documentação executável. Se o teste passa, o comportamento está documentado e garantido.

🔎 2. As Dimensões do Teste (Manual vs Automatizado)

Para entender testes, precisamos olhar para três dimensões diferentes. Elas não se excluem; elas se complementam.

⚙️ Dimensão 1: COMO o teste é executado

Esta dimensão foca no executor da ação.

Critério Teste Manual Teste Automatizado
Executor Ser Humano (QA ou Dev) Script/Ferramenta (Jest, Pytest)
Velocidade Lenta (limitada pela ação humana) Ultra-rápida (milhares por segundo)
Custo de Repetição Alto (exige tempo toda vez) Quase zero (basta rodar o comando)
Ideal para... UX, UI, Exploração inicial Regressão, Lógica, APIs, Performance

🧩 Dimensão 2: O QUE está sendo testado

Aqui focamos no objetivo da validação.

Tipo Descrição Exemplo Prático
Funcional Valida se a regra de negócio funciona. "O desconto de 10% foi aplicado?"
Não Funcional Valida aspectos técnicos e performance. "O sistema suporta 1000 acessos?"

🔗 Dimensão 3: NÍVEL do teste (Onde testamos)

Nível Foco Complexidade
Unitário Uma única função ou método isolado. Baixa (Testa só a lógica)
Integração A comunicação entre duas ou mais partes. Média (Testa a conexão)
E2E (End-to-End) O fluxo completo do ponto de vista do usuário. Alta (Testa o sistema todo)
Tip

Observação Didática: Um erro comum é achar que "Manual" e "Unitário" são opostos. Na verdade:

  • Manual vs Automatizado é o COMO.
  • Unitário vs Integração é o ONDE.
  • Funcional é o O QUE.
    Você pode fazer um teste Funcional de forma Manual ou Automatizada.

🧮 3. Exemplo Prático 1 — Calculadora (Teste Unitário)

O teste unitário é a base de tudo. Ele valida a lógica pura, sem depender de internet, banco de dados ou arquivos externos.

🤲 Cenário de Teste Manual

Antes de automatizar, imagine como você testaria uma calculadora manualmente:

  1. Você abre a calculadora.
  2. Digita 2.
  3. Clica em +.
  4. Digita 3.
  5. Clica em =.
  6. Olha para o visor e confere se aparece 5.
  • Problema: Se você mudar o código, terá que fazer esses 6 passos novamente. E se tiver 50 funções? Você passará o dia clicando em botões.

🤖 Cenário de Teste Automatizado

📁 Estrutura do Projeto (JS) 

calculadora-js/
 ├── src/calculadora.js
 ├── tests/calculadora.test.js

🟨 Código da Calculadora (src/calculadora.js) 

function somar(a, b) {
  return a + b;
}

function dividir(a, b) {
  if (b === 0) throw new Error("Divisão por zero inválida");
  return a / b;
}

module.exports = { somar, dividir };

✅ Criando Testes Bons vs Ruins

❌ O Teste "Preguiçoso" (Ruim):

test('deve retornar um número', () => {
  // RUIM: Se somar(2, 2) retornar 100, ele ainda é um número e o teste passa!
  expect(typeof somar(2, 2)).toBe('number');
});

✔️ O Teste "Engenheiro" (Bom):

const { somar, dividir } = require('../src/calculadora');

test('deve somar 2 + 3 e retornar exatamente 5', () => {
  expect(somar(2, 3)).toBe(5);
});

test('deve lançar erro ao tentar dividir por zero', () => {
  expect(() => dividir(10, 0)).toThrow("Divisão por zero");
});

🟦 Python (Mesma lógica com Pytest) 

# src/calculadora.py
def somar(a, b):
    return a + b

def dividir(a, b):
    if b == 0:
        raise ValueError("Divisão por zero inválida")
    return a / b

# tests/test_calculadora.py
import pytest
from src.calculadora import somar, dividir

def test_somar_corretamente():
    assert somar(2, 3) == 5

def test_divisao_por_zero():
    with pytest.raises(ValueError, match="Divisão por zero inválida"):
        dividir(10, 0)
Important

Observação Didática: O momento de "ouro" do teste é quando você quebra o código de propósito. Mude a função de soma para return a * b e rode o teste. Se o teste falhar, ele é um bom teste. Se ele continuar passando (como o teste "preguiçoso"), ele é inútil.

🌐 4. Exemplo Prático 2 — API de Produtos (Integração)

Nos testes de integração, validamos se a nossa API "fala" corretamente com as outras camadas (dados, roteamento, etc).

🤲 Cenário de Teste Manual

Para testar uma API manualmente:

  1. Você inicia o servidor.
  2. Abre o Postman ou o navegador em http://localhost:3000/produtos.
  3. Lê o JSON retornado.
  4. Confere visualmente se o "Notebook" está lá e se o preço é 3000.
  • Risco: Você pode esquecer de olhar um campo, ou não perceber que um ID mudou.

🤖 Cenário de Teste Automatizado

🟨 API em Node.js (Express) 

const express = require('express');
const app = express();

const produtos = [
  { id: 1, nome: "Notebook", preco: 3000 },
  { id: 2, nome: "Mouse", preco: 150 }
];

app.get('/produtos', (req, res) => {
  res.json(produtos);
});

module.exports = app;

✅ Validação Real (Testes de Integração) 

const request = require('supertest');
const app = require('../src/app');

test('GET /produtos deve retornar a lista completa com dados corretos', async () => {
  const res = await request(app).get('/produtos');

  expect(res.statusCode).toBe(200);
  
  // Validação profunda: não apenas o status, mas o CONTEÚDO
  expect(res.body.length).toBe(2);
  expect(res.body[0]).toEqual({
    id: 1,
    nome: "Notebook",
    preco: 3000
  });
});

🟦 Python (Flask + Pytest) 

# app.py
from flask import Flask, jsonify

app = Flask(__name__)
produtos = [{"id": 1, "nome": "Notebook", "preco": 3000}]

@app.route('/produtos')
def listar():
    return jsonify(produtos), 200

# test_app.py
def test_listar_produtos(client): # 'client' é uma fixture do pytest-flask
    res = client.get('/produtos')
    assert res.status_code == 200
    assert res.json[0]['nome'] == "Notebook"
Note

Observação Didática: Testar API não é só ver se o servidor "está vivo" (Status 200). É garantir que o contrato de dados foi cumprido. Se o front-end espera preco e você retorna valor, o sistema quebra, mesmo com Status 200.

🚀 5. Resumo de Boas Práticas

O que fazer (✅) O que evitar (❌)
Criar testes que falham se o código mudar. Criar testes que passam "não importa o quê".
Testar comportamentos (o que o código faz). Testar a implementação (como ele faz).
Manter testes rápidos e independentes. Fazer um teste depender do resultado de outro.
Nomear os testes de forma clara (ex: "deve somar..."). Usar nomes genéricos (ex: "teste 1").

🎯 6. Conclusão e Desafio

Testes automatizados não são sobre "fazer mais código", mas sim sobre ter liberdade. Quando você tem uma boa suite de testes, você pode refatorar seu sistema com a confiança de que será avisado imediatamente se algo parar de funcionar.

🏆 Desafio Final

  1. Pegue o exemplo da calculadora.
  2. Adicione uma função de subtrair.
  3. Crie um teste manual (tente fazer a conta na cabeça ou papel).
  4. Crie o teste automatizado correspondente.
  5. Provoque um erro: faça a subtração retornar a soma e veja seu teste "gritar" no terminal.
