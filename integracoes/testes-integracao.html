Testes de Integração: Práticas e Validação de Dados (JavaScript e Python)

1. Introdução: O que são Testes de Integração?

Testes de integração garantem que diferentes partes do seu sistema (como o banco de dados, a API e as regras de negócio) funcionem corretamente quando combinadas. Diferente dos testes unitários, que isolam pequenos pedaços de código, a integração testa o fluxo completo de uma funcionalidade.

2. Configurações Iniciais

Para criar testes de integração consistentes, precisaremos de ferramentas para simular chamadas HTTP e validar respostas.

JavaScript (Node.js + Jest + Express)

Em Node.js, a combinação mais comum para testar APIs Express é o Jest com o Supertest.

# Iniciar o projeto
npm init -y

# Instalar Express para criar a API
npm i express

# Instalar Jest e Supertest como dependências de desenvolvimento
npm i -D jest supertest

Adicione ao seu package.json:

"scripts": {
  "test": "jest",
  "test:watch": "jest --watchAll"
}

Python (Flask + Pytest)

Em Python, podemos usar o pytest para rodar os testes e o cliente de teste embutido do próprio Flask para fazer as requisições.

# Instalar dependências
pip install flask pytest

3. Testes de Integração: Rotas e Validação

Exemplo de Teste SEM Validação (Evite isso!)

Frequentemente desenvolvedores criam uma rota, mandam a requisição e verificam apenas um fato isolado:

JavaScript:

test('deve listar produtos (RUIM)', async () => {
  const response = await request(app).get('/produtos');
  
  // RUIM: Se a rota retornar um array vazio ou não trouxer os atributos corretos,
  // esse teste vai passar e a API pode estar quebrada!
  expect(response.status).toBe(200); 
});

Exemplo Correto: Simulando Rota com Validação Completa

JavaScript (Express + Supertest)

Estrutura de pastas:

/src
├── produtos.js
├── app.js
└── app.test.js

produtos.js (Banco de Dados Falso)

const produtos = [
  { id: 1, nome: "Notebook", preco: 3500 },
  { id: 2, nome: "Mouse", preco: 150 },
];

module.exports = { produtos };

app.js (Sua API)

const express = require('express');
const { produtos } = require('./produtos');

const app = express();
app.use(express.json());

app.get('/produtos', (req, res) => {
  res.status(200).json(produtos);
});

module.exports = app;

app.test.js (Seus Testes)

const request = require('supertest');
const app = require('./app');

describe('GET /produtos', () => {
  test('deve retornar lista de produtos perfeitamente igual', async () => {
    const response = await request(app).get('/produtos');

    // Validação da Camada HTTP
    expect(response.status).toBe(200);

    // Validação Ativa do Corpo da Resposta (Deep Equality)
    expect(response.body).toEqual([
      { id: 1, nome: "Notebook", preco: 3500 },
      { id: 2, nome: "Mouse", preco: 150 },
    ]);
  });

  test('cada produto deve ter os atributos corretos', async () => {
    const response = await request(app).get('/produtos');
    const primeiroProduto = response.body[0];

    // Validação ativa que garante que o esqueleto do retorno não perde dados!
    expect(primeiroProduto).toHaveProperty('id');
    expect(primeiroProduto).toHaveProperty('nome');
    expect(primeiroProduto).toHaveProperty('preco');
  });
});

Executando os testes:

npm test

Python (Flask + Pytest)

Estrutura de pastas:

/src
├── produtos.py
├── app.py
└── test_app.py

produtos.py (Banco de Dados Falso)

produtos_db = [
    {"id": 1, "nome": "Notebook", "preco": 3500},
    {"id": 2, "nome": "Mouse", "preco": 150},
]

app.py (Sua API)

from flask import Flask, jsonify, request
from produtos import produtos_db

app = Flask(__name__)

@app.route('/produtos', methods=['GET'])
def listar_produtos():
    return jsonify(produtos_db), 200

if __name__ == '__main__':
    app.run()

test_app.py (Seus Testes)

import pytest
from app import app

@pytest.fixture
def client():
    # Cliente de teste providenciado pelo Flask
    with app.test_client() as client:
        yield client

def test_listar_produtos(client):
    resposta = client.get('/produtos')
    
    # Validação da Camada HTTP
    assert resposta.status_code == 200
    
    # Validação Ativa do Corpo da Resposta
    dados = resposta.get_json()
    assert isinstance(dados, list)
    assert len(dados) == 2
    
    # Validando estrutura de retorno do primeiro item
    primeiro_produto = dados[0]
    assert 'id' in primeiro_produto
    assert 'nome' in primeiro_produto
    assert 'preco' in primeiro_produto
    assert primeiro_produto['nome'] == "Notebook"

Executando os testes:

pytest

Prática: Criando e Testando uma Nova Rota

Com base na API de produtos demonstrada acima, tente fazer o seguinte exercício na sua máquina (escolhendo JavaScript ou Python):

  1. Nova Rota: Adicione uma nova rota GET /produtos/:id (ou /produtos/<int:id> no Flask) no seu servidor HTTP. Esta rota deve ler o ID passado na URL, buscar o produto específico na sua lista (produtos.js ou produtos.py) e retorná-lo.
  2. Teste de Integração: Crie um novo caso de teste no seu arquivo de testes (app.test.js ou test_app.py). Use o seu simulador de cliente HTTP (Supertest ou cliente do Flask) para chamar a nova rota, solicitando, por exemplo, o produto de ID 1.
  3. Checagem de Sucesso: Faça a asserção verificando se o status HTTP recebido na resposta é igual a 200 (OK).

Observação sobre Validação de Dados:
A validação de dados dentro de testes de integração exige verificar ativamente as entradas e saídas da sua API ou serviço. Testar não é apenas garantir que uma rota responda sem quebrar (como visto no exemplo "RUIM"); testar corretamente significa atestar se o comportamento e o formato dos dados recebidos correspondem exatamente ao que as regras de negócio preveem. Quando ignoramos essas validações, criamos testes superficiais que geram uma "falsa segurança": o teste passa com um Status 200, mas a resposta pode estar vazia ou incorreta.

4. Teste de Erros e Falhas Válidos (Caminhos Tristes)

Nem só de acertos vivem os testes de integração. Garantir que a sua aplicação responde adequadamente — e falha de maneira controlada — quando lida com cenários indesejados é crucial.

Quando configuramos os testes para esperar essas falhas (como um Status HTTP 400 ou 404), os testes passam de forma bem-sucedida pois confirmamos que o sistema bloqueou as anomalias corretamente.

A. Faltando Dados Obrigatórios (Validação de Payload)

Ao construir rotas via POST, sua API deve barrar chamadas onde atributos primordiais estiverem ausentes da requisição.

JavaScript:

// app.test.js
test('deve retornar status 400 (Bad Request) ao processar um produto faltando atributos vitais', async () => {
    // Enviando requisição sem colocar a propriedade 'nome'
    const payloadIncompleto = { preco: 150 }; 

    const response = await request(app)
        .post('/produtos')
        .send(payloadIncompleto);

    // Validação correta: esperando pelo erro padronizado da API
    expect(response.status).toBe(400);
    expect(response.body).toHaveProperty("erro");
    expect(response.body.erro).toBe("O atributo 'nome' é obrigatório.");
});

Python:

# test_app.py
def test_criar_produto_incompleto(client):
    # Enviando sem o atributo 'nome'
    payload_incompleto = {"preco": 150}
    
    resposta = client.post('/produtos', json=payload_incompleto)
    
    assert resposta.status_code == 400
    dados = resposta.get_json()
    assert 'erro' in dados
    assert "obrigatório" in dados['erro'].lower()

B. Resolução Pacífica de Buscas Não Encontradas (404 Not Found)

Teste também situações onde o usuário busca por algo que logicamente não existe no banco de dados. A API não deve "quebrar", mas sim devolver um erro 404 formatado.

JavaScript:

test('deve retornar status 404 (Not Found) para produto inexistente', async () => {
    const idInexistente = 999999;
    
    const response = await request(app).get(`/produtos/${idInexistente}`);

    // Confirma se a API lidou de boa forma enviando 404
    expect(response.status).toBe(404);
    expect(response.body).toEqual({ erro: "Produto não encontrado" });
});

Python:

def test_produto_nao_encontrado(client):
    id_inexistente = 999999
    
    resposta = client.get(f'/produtos/{id_inexistente}')
    
    assert resposta.status_code == 404
    dados = resposta.get_json()
    assert dados == {"erro": "Produto não encontrado"}

Prática: Aprofundando nas Validações (Testes dos Caminhos Tristes)

Agora que você já sabe como verificar se a API retorna 200 em caso de sucesso na rota de detalhe do produto, desafie-se a testar os cenários de falha para essa rota específica:

  1. Erro de Validação de ID: Na rota GET /produtos/:id, o ID é um número? Adicione um teste que envie um ID inválido (por exemplo, uma string como "abc" ou um ID negativo) e valide se a API retorna o status 400 (Bad Request) com uma mensagem de erro apropriada.
  2. Produto Não Encontrado: Crie um teste para um ID que sabemos que não existe na lista (ex: 9999). Valide se a resposta é 404 (Not Found).

5. Cobertura de Código em Testes de Integração

A cobertura de código (Code Coverage) é uma métrica que indica qual porcentagem do código-fonte do seu projeto foi executada durante a rodada de testes. Em testes de integração, a cobertura é vital para garantir que não apenas os caminhos "felizes" da API estão sendo tocados, mas também os tratamentos de erro, validações e middlewares.

Como Medir a Cobertura?

JavaScript (Jest)
O Jest já possui a ferramenta de cobertura embutida. Basta rodar com a flag --coverage:

npx jest --coverage
# Ou, se adicionado ao package.json:
npm run test:coverage

Isso gerará uma tabela no terminal mostrando quais linhas das suas rotas e arquivos de serviço não foram executadas por nenhum teste.

Python (pytest)
Em Python, precisamos da biblioteca pytest-cov:

pip install pytest-cov
pytest --cov=app # 'app' é o nome do módulo a ser analisado

O Que a Cobertura Mostra?

  • Statements/Linhas: Quantas linhas de código foram lidas.
  • Branches (Ramificações): Garante que todos os ifs, elses e try/catches da sua API foram exercitados (por exemplo, validou tanto o erro 400 quanto o sucesso 200).
  • Functions: Quantas funções/controladores foram chamados.

Atenção: Ter 100% de cobertura não significa que a API não possui bugs; significa apenas que cada linha foi lida. Sem os "Testes de Validação" ativos, como vimos antes, a cobertura pode ser uma falsa métrica de segurança.

6. Boas Práticas (Mentalidade de Testes)

  1. Teste comportamentos e retornos, não a implementação: Não avaliamos como o programa está executando os processos internos, apenas confirmamos se a resposta HTTP e o JSON resultante obedecem aos acordos da API.
  2. Sempre inclua "edge cases" (Cenários extremos): Pense além do fluxo perfeito (Caminho Feliz). Teste listas vazias, envio de dados em branco, tipagens erradas (enviar texto onde deveria ser número) e requisições para IDs que não existem.
  3. Mantenha o Banco Limpo: Em projetos reais, garanta que seu banco de dados de testes seja resetado ou limpo antes de cada teste de integração (utilizando beforeEach no Jest ou fixtures no Pytest) para evitar que um teste influencie o outro.

7. Desafio Prático: Testando uma API de Biblioteca (Livros)

Objetivo: Implementar os testes de integração para uma API de Livros de uma biblioteca. Escolha a sua linguagem preferida (JavaScript ou Python).

Imagine que você tem uma API com as seguintes rotas:

  • POST /livros - Cadastra um livro (espera os campos nome, autor e genero e cria internamente um id).
  • GET /livros/:id - Busca um livro pelo seu id.

O que você deve desenvolver no arquivo de testes:

  1. Teste de Caminho Feliz: Crie um teste que chame o POST /livros com dados válidos e garanta que ele retorna o Status 201 e os dados do livro recém-cadastrado (agora contendo seu id).
  2. Teste de Validação de Dados: Crie um teste POST /livros enviando um objeto sem o campo autor. Valide se a API retorna Status 400 e uma mensagem de erro adequada.
  3. Teste de Caminho Triste: Crie um teste GET /livros/:id buscando um ID de livro não cadastrado. Valide se a resposta é um Status 404.

Dica: Lembre-se de validar ativamente os objetos de resposta (response.body em JS ou resposta.get_json() em Python), verificando se as chaves corretas (id, nome, autor e genero) existem no retorno!
