Plano de Testes de Software

1. Definição e Objetivos

O Plano de Testes é um documento estratégico que descreve o escopo (o que será testado), a abordagem (como será testado), os recursos e o cronograma das atividades de teste. Seu objetivo principal é fornecer um roteiro claro para a execução do controle de qualidade em um projeto de software.

Por que o Plano de Testes é tão importante?

  • Guia Definitivo: Ele responde às perguntas fundamentais: O que testar? Como testar? Quando testar? Quem será responsável?
  • Documento Vivo: O Plano de Testes não deve ser escrito e esquecido. Ele deve evoluir junto com o software. Se os requisitos mudarem, o plano também deve ser atualizado.
  • Garantia de Qualidade Sustentável: Ajuda a prevenir defeitos antes que fiquem muito caros para serem corrigidos, garantindo a entrega de um produto confiável.
  • Alinhamento com Stakeholders (Partes Interessadas): Garante que desenvolvedores, QAs (Analistas de Qualidade), Product Owners (POs - Donos do Produto), gerentes e clientes tenham clareza sobre os critérios de aceitação (o que precisa funcionar para o software ser aprovado).

2. Conteúdo Essencial (Baseado na norma ISO/IEC/IEEE 29119-3)

Um plano de testes robusto deve seguir padrões da indústria, como a norma ISO 29119, que define como documentar testes de software. Abaixo, exploramos os componentes essenciais:

2.1 Visão Geral e Escopo

Define claramente os limites do que será testado.

  • O que será testado (In-Scope): Quais módulos, funcionalidades, APIs (interfaces de comunicação entre sistemas) ou telas serão validados.
  • O que NÃO será testado (Out-of-Scope): Essencial para gerenciar as expectativas. Por exemplo, você pode decidir que testes de estresse (testar com muitos usuários ao mesmo tempo) não farão parte desta etapa.
  • Dependências Externas: Identificação de serviços de terceiros que afetam o teste (ex: um sistema externo de pagamento ou envio de SMS).

2.2 Estratégia de Teste

A espinha dorsal de como os testes serão feitos na prática.

  • Tipos de Teste: Definição se os testes focarão em aspectos funcionais (o software faz o que deveria fazer?), testes de regressão (uma nova mudança quebrou algo que já funcionava?) ou não-funcionais (como performance, segurança ou facilidade de uso).
  • Níveis de Teste: As etapas de validação: começam pelo Teste Unitário (testar partes isoladas do código), passam para a Integração (testar se as partes se comunicam bem) e vão até o E2E - End To End (teste ponta a ponta, simulando um usuário real do início ao fim).
  • Manual vs. Automatizado: Definir o que será feito por um testador humano (testes manuais) e o que será feito por scripts de código (testes automatizados).

2.3 Equipe e Responsabilidades

Define quem faz o quê no ciclo de validação.

  • Papéis: Quem planeja e testa (QA), quem corrige os defeitos (Desenvolvedor) e quem dá o aceite final (Product Owner).
  • Matriz de Responsabilidades (Ex: RACI): Uma tabela que define quem é o Responsável por executar, quem é a Autoridade para aprovar, quem deve ser Consultado e quem deve ser Informado sobre os testes.

2.4 Riscos e Contingências

Todo projeto possui riscos. Prever problemas evita o pânico.

  • Exemplos de Riscos: O servidor de testes cair frequentemente, uma ferramenta parar de funcionar ou o prazo ficar muito curto.
  • Planos de Contingência ("Plano B"): O que a equipe fará se o risco acontecer? Por exemplo: se a API externa cair, podemos usar "Mocks" (simulações falsas da API) para continuar testando.

2.5 Cronograma e Estimativas

A previsão do tempo e dos recursos.

  • Fases e Etapas: Definir quando o plano será escrito, quando os testes serão desenvolvidos e quando serão executados.
  • Estimativas de Tempo: Quantas horas ou dias a equipe precisará para finalizar os testes sem atrasar a entrega.

2.6 Ferramentas e Ambientes de Teste

Onde e com o que os testes serão feitos.

  • Ambientes de Teste: Os testes nunca devem ser feitos no banco de dados real dos clientes (Produção). Eles são feitos em ambientes separados: "Local" (no computador do desenvolvedor) e "Staging/Homologação" (um servidor de testes que imita a Produção).
  • Massa de Teste (Dados de Teste): Dados falsos usados para testar (ex: e-mails fictícios). Eles devem seguir leis de privacidade, como a LGPD, garantindo que nenhum dado real de cliente seja exposto.

3. Arquitetura e os Níveis de Escopo

Planos de teste podem ser divididos dependendo do tamanho do projeto:

3.1 Plano Master (Plano de Testes Mestre - PTM)

  • Oferece uma visão global. É usado quando o sistema é muito grande (ex: tem um aplicativo mobile, um site web e vários serviços na nuvem ao mesmo tempo). Ele documenta as regras gerais de como tudo será testado em conjunto.

3.2 Planos Específicos

São documentos menores e focados em partes específicas:

  • Plano de Teste de Unidade: Focado apenas no código dos desenvolvedores.
  • Plano de Teste de Integração (APIs): Focado em testar a comunicação entre os sistemas e como eles lidam com erros.
  • Plano de Segurança e Performance: Focado apenas em tentar invadir o sistema (Pen-Test) ou derrubá-lo com muitos acessos (Stress Test).

4. O Impacto das Metodologias de Trabalho (Ágil vs Tradicional)

A forma como a equipe de tecnologia trabalha muda completamente o andamento dos testes.

4.1 Metodologia Tradicional (Cascata/Waterfall)

  • O plano é escrito de forma extensa e muito detalhada meses antes de o código começar a ser feito.
  • Os testes só ocorrem no final do projeto. Isso é arriscado, pois descobrir um defeito grave no final custa muito caro e atrasa o lançamento.

4.2 Metodologia Ágil (Scrum / Kanban)

  • Os planos são curtos, diretos e atualizados constantemente (geralmente a cada 1 ou 2 semanas, nas chamadas Sprints).
  • Testes desde o início ("Shift-Left Testing"): O QA trabalha junto com a equipe desde o planejamento da funcionalidade, evitando que erros sejam sequer codificados.
  • Feedback Rápido: Qualquer erro é descoberto em poucas horas através de testes automatizados executados por robôs (CI/CD - Integração Contínua), impedindo que o problema chegue ao cliente.

5. Técnicas de Teste (Como criar os cenários)

São técnicas mentais para saber exatamente o que testar sem perder tempo com milhares de tentativas inúteis.

5.1 Teste de Caixa Preta (Black-Box)

  • O Enfoque: O testador não olha para o código-fonte. Ele apenas interage com a tela ou sistema como um usuário normal ("Coloco esse dado aqui, clico aqui, e vejo o que acontece").

5.2 Teste de Caixa Branca (White-Box)

  • O Enfoque: O testador tem acesso ao código-fonte e testa a lógica interna (os caminhos matemáticos e os "IFs" e "ELSEs" do código). É focado em garantir que todas as linhas de código foram validadas.

5.3 Partição de Equivalência

  • O Enfoque: Evita testar todas as possibilidades do mundo dividindo os dados em "grupos equivalentes".
  • Exemplo prático: Se um sistema só aceita idades entre 10 e 50 anos, você não precisa testar os números 11, 12, 13, 14... Basta criar 3 grupos: um número menor que 10 (rejeitado), um número entre 10 e 50 (aceito) e um número maior que 50 (rejeitado). Você faz 3 testes em vez de dezenas.

5.4 Análise do Valor Limite (BVA - Boundary Value Analysis)

  • O Enfoque: A maioria dos erros de programação acontece nos "limites" das regras. Essa técnica foca em testar exatamente as bordas numéricas.
  • Exemplo prático: Usando a regra de 10 a 50 anos, testaremos as fronteiras exatas: 9 (falha), 10 (sucesso), 50 (sucesso) e 51 (falha).

5.5 Teste Exploratório (Ad-hoc)

  • O Enfoque: Testar o sistema livremente, sem um roteiro engessado, usando a intuição, criatividade e a experiência do testador para achar erros onde os testes normais e automatizados não alcançam.

6. Ferramentas Comuns no Mercado

  • Documentação (Onde escrever o plano de testes): Jira/Confluence, Notion, X-Ray, Qase.io.
  • Automação de Testes (Para escrever os scripts lógicos): Jest (para JavaScript), Cypress ou Playwright (para simular cliques na web de forma automática), Postman (para testar comunicação de APIs).

7. Critérios de Qualidade (DoR e DoD)

Em equipes profissionais, um sistema não vai para os usuários reais "só porque achamos que está bom". Precisamos de aprovações formais (portões de qualidade):

7.1 Critérios de Entrada (DoR - Definition of Ready)

  • O que precisamos ter ANTES de começar a testar? (Exemplo: O código deve estar pronto, o ambiente de testes deve estar online sem quedas e o design da tela deve ter sido aprovado).

7.2 Critérios de Saída/Conclusão (DoD - Definition of Done)

  • Quando podemos dizer que "Terminamos" os testes? (Exemplo: 100% dos testes críticos passaram com sucesso, e não existe nenhum bug impeditivo aberto na plataforma).

7.3 KPIs (Métricas de Desempenho)

  • Cobertura de Código (Code Coverage): Qual porcentagem do código dos programadores foi efetivamente testada (um bom número é acima de 80%).
  • Matriz de Rastreabilidade: Uma tabela que comprova que todos os requisitos pedidos pelo cliente têm pelo menos um teste feito garantindo seu funcionamento.

7.4 Gestão de Defeitos (Bugs)

Quando achamos um bug, precisamos classificá-lo para que a equipe saiba a urgência:

  • Severidade (O impacto técnico): Quão grave é o erro para o sistema? (Ex: O sistema travar completamente no fechamento do carrinho de compras tem Severidade Altíssima. Um botão ligeiramente torto tem Severidade Baixa).
  • Prioridade (O impacto no negócio): Quão rápido precisa ser corrigido pela equipe? (Ex: Um palavrão escrito na página inicial da empresa pode ter Severidade técnica Baixa (não quebra o sistema), mas Prioridade Extrema (afeta a reputação da empresa e deve ser corrigido imediatamente)).

8. Exemplo Prático: Calculadora em JavaScript

Para não ficarmos apenas na teoria, vamos aplicar os conceitos acima em um pequeno módulo de sistema: uma Calculadora Digital.

⚙ 8.1 O Código da Aplicação (A Classe Lógica)

Abaixo temos a classe Calculadora em JavaScript. É esse código que vamos submeter aos testes mapeados na seção seguinte.

class Calculadora {
    somar(a, b) {
        return a + b;
    }

    subtrair(a, b) {
        return a - b;
    }

    multiplicar(a, b) {
        return a * b;
    }

    dividir(a, b) {
        // Validação de tipo de dados (Se não for número, dispara erro)
        if (typeof a !== 'number' || typeof b !== 'number') {
            throw new TypeError("Parâmetros inválidos. O sistema aceita apenas números inteiros.");
        }

        // Validação de regra de negócio (Divisão por zero)
        if (b === 0) {
            throw new Error("A divisão por zero não é permitida.");
        }

        return a / b;
    }
}

8.2 Aplicando as Técnicas de Teste na Calculadora

  • Caminho Feliz (Happy Path): Testar a rotina de sucesso, onde o usuário envia dados normais e tudo dá certo.
    • Exemplo: Somar 2 + 2 e receber 4.
  • Caminho Triste (Sad Path / Caminhos Negativos): Testar situações imprevisíveis ou incorretas para ver se o sistema sabe se defender sem quebrar.
    • Exemplo: Tentar dividir 10 por 0 (Zero). O sistema deve nos avisar de forma controlada que isso é inválido, e não simplesmente travar.
    • Exemplo: Tentar enviar uma letra (ex: "Dois") em vez de um número. O sistema deve rejeitar a entrada exigindo um número.

Aplicando a Teoria (Seção 5) na Prática:

  • Ao testar a divisão por zero, estamos usando a Análise de Valor Limite (BVA), pois o zero é uma fronteira matemática crítica.
  • Ao enviar letras ("Dois") em vez de números, estamos aplicando a Partição de Equivalência, validando o grupo inteiro de "dados de tipos inválidos" sem precisar testar todas as letras do alfabeto.

🤖 8.3 Automação dos Testes (Utilizando Jest)

Com o plano de testes definido (que veremos na Seção 9), a equipe de automação escreve os scripts em Jest para executar as validações continuamente. Repare como cada teste reflete diretamente um ID (CT-01, CT-02...) da nossa tabela de testes.

// Pré-requisito: npm install -D jest
// Comando para rodar: npm run test

const Calculadora = require('./calculadora'); 

describe("Plano de Testes - Calculadora", () => {
    let calc;

    beforeEach(() => {
        // Preparação: Instancia a calculadora antes de cada teste
        calc = new Calculadora();
    });

    test("CT-01 - Validar a soma de dois números inteiros positivos", () => {
        expect(calc.somar(2, 2)).toBe(4);
    });

    test("CT-02 - Validar a subtração com resultado positivo", () => {
        expect(calc.subtrair(5, 3)).toBe(2);
    });

    test("CT-03 - Validar a multiplicação de dois números inteiros", () => {
        expect(calc.multiplicar(3, 4)).toBe(12);
    });

    test("CT-04 - Validar a divisão exata entre dois números", () => {
        expect(calc.dividir(10, 2)).toBe(5);
    });

    test("CT-05 - Validar o erro ao tentar dividir por zero", () => {
        expect(() => calc.dividir(10, 0)).toThrow("A divisão por zero não é permitida.");
    });

    test("CT-06 - Validar a proteção contra tipos de dados não numéricos", () => {
        // Simulando a falha: O teste exige uma mensagem exata, mas o código retorna outra (TypeError).
        // Isso fará o teste falhar, conforme apontado no relatório da Seção 9.
        expect(() => calc.dividir("Dois", 2)).toThrow("Ambos os parâmetros devem ser números.");
    });

    test.skip("CT-07 - Validar a proteção ao enviar um valor nulo", () => {
        // Teste bloqueado: Utilizamos o .skip() do Jest para ignorar este teste por enquanto.
        expect(() => calc.dividir(null, 5)).toThrow(TypeError);
    });
});

Desafio Prático: O Sistema de Notas Escolares

Como exercício de fixação, chegou a sua vez de atuar como o Analista de Qualidade (QA) de um projeto. A sua equipe de desenvolvimento acabou de criar um pequeno módulo em JavaScript chamado SistemaAlunos.

A regra de negócio solicitada pelo cliente foi:

  1. O sistema deve calcular a média aritmética de 4 notas.
  2. Com base nessa média, o sistema deve classificar a situação escolar do aluno:
    • Aprovado: Média maior ou igual a 7.
    • Recuperação: Média maior ou igual a 5 e menor que 7.
    • Reprovado: Média menor que 5.

Aqui está o código desenvolvido pela sua equipe:

O Código Principal (sistemaAlunos.js) 

class SistemaAlunos {
    calcularMedia(n1, n2, n3, n4) {
        if (
            typeof n1 !== 'number' || typeof n2 !== 'number' || 
            typeof n3 !== 'number' || typeof n4 !== 'number'
        ) {
            throw new TypeError("Todas as notas devem ser parâmetros numéricos.");
        }

        const notas = [n1, n2, n3, n4];
        for (let nota of notas) {
            if (nota < 0 || nota > 10) {
                throw new Error("As notas devem estar entre 0 e 10.");
            }
        }

        const media = (n1 + n2 + n3 + n4) / 4;
        return Number(media.toFixed(2));
    }

    verificarStatus(media) {
        if (typeof media !== 'number' || media < 0 || media > 10) {
            throw new Error("A média informada é inválida.");
        }

        if (media >= 7.0) {
            return "Aprovado";
        } else if (media >= 5.0) {
            return "Recuperação";
        } else {
            return "Reprovado";
        }
    }
}

O Início da Bateria de Testes (sistemaAlunos.test.js)

A equipe de engenheiros mapeou os testes básicos automatizados com Jest focando apenas nos fluxos positivos (Caminho Feliz), mas você precisa completar a cobertura:

const SistemaAlunos = require('./sistemaAlunos');

describe("Bateria de Testes - Sistema de Alunos", () => {
    let sistema;

    beforeEach(() => {
        sistema = new SistemaAlunos();
    });

    test("CT-01 - Deve calcular a média corretamente de quatro notas válidas", () => {
        expect(sistema.calcularMedia(7, 8, 9, 6)).toBe(7.5);
    });

    test("CT-02 - Deve retornar 'Aprovado' para media exata igual 7.0", () => {
        expect(sistema.verificarStatus(7.0)).toBe("Aprovado");
    });

    test("CT-03 - Deve retornar 'Recuperação' para média >= 5 e < 7", () => {
        expect(sistema.verificarStatus(6.5)).toBe("Recuperação");
    });

    test("CT-04 - Deve retornar 'Reprovado' para média < 5", () => {
        expect(sistema.verificarStatus(4.9)).toBe("Reprovado");
    });
});

📋 Mão na Massa: O Seu Desafio!

Sua tarefa consiste em elaborar o Plano de Testes (na forma documentada ou em tabela de CTs) contemplando todas as ramificações e situações limítrofes para garantir a segurança contra erros desse projeto.

Passo a passo esperado:

  1. Leia a lógica do código e identifique as Exceções (Caminhos Negativos). O que deverá acontecer se enviarmos uma letra na nota? Ou se inserirmos a nota 10.1? E a nota -2?
  2. Escreva os cenários de teste aplicando a técnica de Análise do Valor Limite (BVA), focando exatamente nos números das bordas das regras: 4.9, 5.0, 6.9, 7.0, 10.0 e 10.1.
  3. Descreva no plano como você usou a técnica de Partição de Equivalência para agrupar as notas inválidas.
  4. (Avançado / Opcional): Construa os scripts automatizados usando expect().toThrow() em Jest para simular as rotinas de erro mapeadas acima.

Desafio Prático 2: Testes de Integração e APIs (Livraria)

Agora que você dominou os testes unitários e lógicos da calculadora e das notas, vamos dar um passo adiante. Sua equipe backend acabou de desenvolver uma nova API Restful para Gestão de Livros.

A regra de negócio solicitada pelo cliente e as rotas disponíveis são as seguintes:

Estrutura do Objeto (O Livro)

Quando um usuário for cadastrar um livro via API, ele deve enviar os seguintes dados:

  • id: (Número) Gerado automaticamente pelo banco de dados.
  • livro: (Texto/String) O título do livro. Obrigatório.
  • autor: (Texto/String) O nome do autor principal. Obrigatório.
  • co_autores: (Array/Lista de Textos) Uma lista com outros autores. Opcional.
  • editora: (Texto/String) O nome da editora do livro. Obrigatório.

Rotas da API Disponíveis

  • GET /livros - Retorna a lista de todos os livros cadastrados.
  • GET /livros/:id - Retorna os detalhes de um livro específico pelo seu ID.
  • POST /livros - Cadastra um novo livro recebendo os dados em formato JSON. (Retorna o status 201 Created em caso de sucesso).

O Código da API (app.js)

Abaixo está o código-fonte desenvolvido pela equipe usando Node.js e o framework Express. Repare que existem algumas validações básicas, mas será que elas protegem o sistema contra todos os problemas?

    res.status(200).json(livros);
});

app.get('/livros/:id', (req, res) => {
    const id = parseInt(req.params.id);
    
    if (isNaN(id)) {
        return res.status(400).json({ erro: "Formato de ID inválido." });
    }
    
    const livro = livros.find(l => l.id === id);
    if (!livro) {
        return res.status(404).json({ erro: "Livro não encontrado." });
    }
    
    res.status(200).json(livro);
});

app.post('/livros', (req, res) => {
    const { livro, autor, co_autores, editora } = req.body;

    // Validações básicas feitas pela equipe
    if (!livro || !autor || !editora) {
        return res.status(400).json({ erro: "Campos obrigatórios ausentes." });
    }

    if (typeof autor !== 'string' || typeof editora !== 'string') {
        return res.status(400).json({ erro: "Autor e editora devem ser textos." });
    }

    const novoLivro = {
        id: proximoId++,
        livro,
        autor,
        co_autores: Array.isArray(co_autores) ? co_autores : [],
        editora
    };

    livros.push(novoLivro);
    res.status(201).json(novoLivro);
});

module.exports = app;
